# OCR エンジン比較 PoC (2026-04-28)

## 結論 (TL;DR)

- **採用エンジン: YomiToku** (Apache-2.0 ベース、商用利用は別ライセンス確認要、個人利用なら可)
- **不採用 (今回省略): PaddleOCR-VL 1.5** — ディスク残量逼迫 (5.7GB) でインストールリスク回避。YomiToku の実測品質が縦書き・横書きとも ◎ で、これ以上の比較は不要と判断
- **リポ構成: B 案 (同リポ別パッケージ別 CLI)** — `kindle_cap` は不変、新たに `book_ocr` パッケージと `book-ocr` CLI を追加。OCR 依存 (~1.5GB) は extras `[ocr]` で隔離

## 目的

`kindle-calibre-tool` の出力を ClaudeCode から「対話的に質問・情報源として活用」するための OCR エンジンを選定し、リポ構成 (A: 同パッケージ混在 / B: 同リポ別パッケージ別 CLI / C: 別リポジトリ) を確定する。

## 候補

| エンジン | 強み | 懸念 |
|---|---|---|
| **YomiToku** | 日本語特化 4 モデル (検出/認識/レイアウト/表)、Markdown ネイティブ、ローカル軽量と評判 | OmniDocBench SOTA ではない、印刷文字のみ、商用利用要確認 |
| **PaddleOCR-VL 1.5** | 2026 SOTA (OmniDocBench 94.50)、5 つの困難ケースに強い | 日本語縦書きは fine-tune 推奨、PaddlePaddle ランタイム重め |

## サンプル

`output/` 配下の実書籍を流用 (新規撮影なし):

- 縦書き: `output/イシューからはじめよ/page_001.png` 〜 `page_005.png`
- 横書き: `output/Kaggleではじめる大規模言語モデル入門/page_001.png` 〜 `page_005.png`

## 環境

- macOS Apple Silicon (Darwin 25.0.0)
- Python 3.12.2 (uv venv)
- 隔離 venv: `experiments/ocr-poc/.venv-yomitoku/`
- 注意: PoC 開始時のディスク残量 8.9GB / 228GB (96% 使用)。YomiToku PoC 後は 5.7GB に減少

### Markdown 出力品質 (目視評価)

| 観点 | 評価 | コメント |
|---|---|---|
| 横書き耐性 | ◎ | 「Natural Language Processing; NLP」のような英語混在も正確 |
| 縦書き耐性 | ◎ | 漢字・送り仮名・読み順すべて正確 |
| 段組保持 | ◎ | 段落改行を `<br>` で保持、Markdown として正しい |
| 見出し検出 | ◎ | 「はじめに」「優れた知的生産に共通すること」を `#` で見出し化 |
| ノイズ | △ | 各ページ上部に「Kindle」+ 書籍名タイトルがメタ反復 → CLI `--ignore_meta` で除外可 |

縦書き本文サンプル (page_003 抜粋):

```text
# 優れた知的生産に共通すること

はじめに

僕がこれまでに見てきた「圧倒的に生産性の高い人」にひと<br>つ共通していることがある。それは、彼らが「ひとつのことを<br>やるスピードが10倍、20倍と速いわけではない」ということだ。
```

横書き本文サンプル (page_005 抜粋):

```text
# はじめに

近年、自然言語処理\(Natural Language Processing; NLP\) は目覚ましい進歩を遂げており、その<br>中核を担う大規模言語モデル\(Large Language Model; LLM\) の登場は、さまざまな革新をもたらし<br>ました。
```

### CLI コマンド

```bash
yomitoku <png_dir_or_file> -f md -o <out_dir> -d mps --reading_order auto [--ignore_meta] [--ignore_ruby]
```

主要オプション:

- `-f md`: Markdown 出力
- `-d mps|cpu|cuda`: デバイス
- `--reading_order auto|left2right|top2bottom|right2left`: 縦書き対応
- `--ignore_meta`: ヘッダー・フッター除外 (本実装では有効化推奨)
- `--ignore_line_break`: 行内改行除外
- `--combine`: 全ページを 1 ファイルに結合
- `-l --lite`: 軽量モデル (未検証)

## PaddleOCR-VL 1.5 (今回省略)

理由: PoC 開始時 8.9GB → YomiToku 後 5.7GB のディスク残量。PaddleOCR-VL は paddlepaddle + モデルで 2-3GB 想定、入れる余地はあるが安全マージンが薄い (容量切れによる環境破壊リスク回避)。YomiToku の品質が ◎ なので追加比較なしで意思決定可能と判断。

将来 PaddleOCR-VL を試したい場合の手順は以下に記録:

```bash
uv venv experiments/ocr-poc/.venv-paddleocr --python 3.12
VIRTUAL_ENV=$(pwd)/experiments/ocr-poc/.venv-paddleocr uv pip install paddleocr paddlepaddle
# CLI: paddleocr の最新ドキュメントは https://www.paddleocr.ai/latest/en/version3.x/pipeline_usage/PaddleOCR-VL.html
```

## A/B/C リポ構成判断

判断ロジック:

| 観測 | 想定 | 実測 | → 推奨 |
|---|---|---|---|
| OCR 依存サイズ | <500MB なら軽 / 500MB-2GB なら中 / >2GB なら重 | **約 1.5GB** (中) | **B または C** |
| OCR ラッパーのコード量 | 100 行未満なら混ぜ可 / 100-300 行なら分離価値あり | **CLI ラッパーで実質 0 行** (subprocess 経由) | **B (関心分離が効く)** |
| 縦書き耐性 | ○ 以上必要 | ◎ | YomiToku 確定 |
| 「Kindle 以外でも使いたい」 | 確定なら C | 設計上は汎用化可、ただし当面は Kindle 用途 | **B (将来 C に昇格できる構造)** |

### 結論: B 案 (同リポ別パッケージ別 CLI)

- `kindle-calibre-tool` リポを継続使用、新たに `src/book_ocr/` パッケージと `book-ocr` CLI を追加
- `kindle_cap` は完全に不変
- pyproject.toml の `[project.optional-dependencies] ocr = [...]` で依存隔離
- パイプライン: `kindle-cap ... && book-ocr output/<book>/`
- 将来 PyPI 単独配布したくなったら別リポに昇格できる構造を保つ

## 再現手順

```bash
git checkout poc/ocr-comparison

# サンプル準備
mkdir -p experiments/ocr-poc/samples-vertical experiments/ocr-poc/samples-horizontal
cp output/イシューからはじめよ/page_00{1,2,3,4,5}.png experiments/ocr-poc/samples-vertical/
cp output/Kaggleではじめる大規模言語モデル入門/page_00{1,2,3,4,5}.png experiments/ocr-poc/samples-horizontal/

# YomiToku セットアップ
uv venv experiments/ocr-poc/.venv-yomitoku --python 3.12
VIRTUAL_ENV=$(pwd)/experiments/ocr-poc/.venv-yomitoku uv pip install yomitoku

# OCR 実行
experiments/ocr-poc/.venv-yomitoku/bin/yomitoku experiments/ocr-poc/samples-vertical \
  -f md -o experiments/ocr-poc/yomitoku-out/vertical -d mps --reading_order auto

experiments/ocr-poc/.venv-yomitoku/bin/yomitoku experiments/ocr-poc/samples-horizontal \
  -f md -o experiments/ocr-poc/yomitoku-out/horizontal -d mps --reading_order auto
```
